# STRUCTURE.md

# Estructura recomendada para una web personal tipo portfolio

Este documento define una estructura genérica para una web personal tipo portfolio. No contiene contenido personal concreto. Sirve como guía de organización, formato, extensión y jerarquía de información.

---

## 1. Objetivo de la web

La web debe permitir que una persona externa entienda rápidamente:

- Quién eres profesionalmente.
- Qué tipo de trabajo haces.
- Qué proyectos has construido.
- Qué habilidades técnicas o profesionales puedes demostrar.
- Cómo contactar contigo.
- Dónde consultar pruebas externas de tu trabajo: GitHub, LinkedIn, CV, publicaciones, demos o repositorios.

La web no debe ser una copia larga del CV. Debe funcionar como una página de presentación con evidencia seleccionada.

---

## 2. Principios generales de estructura

### 2.1. Longitud recomendada

Para una primera versión sólida:

- Una sola página principal es suficiente.
- Entre 5 y 7 secciones.
- Entre 3 y 5 proyectos destacados.
- Entre 600 y 1.200 palabras en total.
- Lectura completa en 3-5 minutos.

### 2.2. Orden recomendado

Estructura base:

```txt
1. Hero / Introducción
2. Sobre mí
3. Proyectos destacados
4. Experiencia / trayectoria
5. Habilidades
6. Contacto
7. Footer
```

Estructura más completa:

```txt
1. Hero / Introducción
2. Sobre mí
3. Áreas de interés
4. Proyectos destacados
5. Experiencia / educación
6. Stack tecnológico / habilidades
7. Reconocimientos / publicaciones / actividades
8. Contacto
9. Footer
```

---

## 3. Navegación superior

### Objetivo

Permitir saltar rápidamente a las secciones principales.

### Formato recomendado

Una barra superior simple:

```txt
Nombre / Marca personal      About   Projects   Experience   Contact
```

### Extensión

- 4 a 6 enlaces como máximo.
- No incluir demasiadas categorías.
- Mantener nombres cortos.

### Apartados recomendados

```txt
About
Projects
Experience
Skills
Contact
```

### Reglas

- La navegación debe estar visible desde el inicio.
- En mobile, puede convertirse en menú compacto.
- No debe ocupar demasiado espacio vertical.
- El nombre o logo debe llevar al inicio de la página.

---

## 4. Hero / sección inicial

### Objetivo

Explicar en 5-10 segundos quién eres y qué haces.

### Contenido recomendado

Debe incluir:

```txt
Nombre
Título profesional o académico
Frase de posicionamiento
Botones principales
Enlaces externos clave
```

### Formato sugerido

```txt
[Nombre]

[Título breve: rol, disciplina o especialización]

[Frase de 1-2 líneas que explique el tipo de problemas, productos o sistemas que construyes.]

[Botón principal: Ver proyectos]
[Botón secundario: Contactar / Descargar CV]

[GitHub] [LinkedIn] [Email]
```

### Extensión recomendada

- Título principal: 3-8 palabras.
- Subtítulo profesional: 6-14 palabras.
- Descripción: 1-2 frases.
- Máximo: 60-90 palabras.

### Ejemplo de estructura genérica

```txt
[Nombre completo]

[Rol o especialización principal]

Construyo [tipo de soluciones] aplicando [áreas de conocimiento] para resolver [tipo de problemas].

[View Projects] [Contact]
```

### Reglas

- Evitar párrafos largos.
- Evitar frases vagas como “apasionado por la tecnología” si no aportan evidencia.
- Debe quedar claro el eje profesional principal.
- Incluir una llamada clara hacia proyectos o contacto.

---

## 5. Sobre mí / About

### Objetivo

Dar contexto personal-profesional sin convertirlo en autobiografía.

### Contenido recomendado

Debe responder:

```txt
¿Qué estudias o a qué te dedicas?
¿Qué tipo de problemas te interesan?
¿Qué tipo de proyectos sueles construir?
¿Qué rasgos definen tu forma de trabajar?
```

### Formato sugerido

```txt
## About

Soy [perfil general] con interés en [áreas principales]. Mi trabajo se centra en [tipo de actividad o construcción], combinando [habilidades o disciplinas].

Me interesa especialmente [área 1], [área 2] y [área 3], con foco en crear soluciones que sean [criterio 1], [criterio 2] y [criterio 3].
```

### Extensión recomendada

- 1-2 párrafos.
- 80-150 palabras.
- No más de 3 ideas principales.

### Reglas

- No repetir todo el CV.
- No listar habilidades aquí; eso va en Skills.
- No escribir una historia completa.
- Debe aportar contexto y dirección profesional.

---

## 6. Áreas de interés / Focus Areas

### Objetivo

Mostrar los ejes temáticos o profesionales de la persona.

### Cuándo incluir esta sección

Incluirla si el perfil combina varias áreas o si los proyectos son diversos.

### Formato recomendado

Tarjetas breves:

```txt
[Área 1]
Descripción de 1 frase.

[Área 2]
Descripción de 1 frase.

[Área 3]
Descripción de 1 frase.
```

### Número recomendado

- 3 áreas como ideal.
- 4 como máximo.

### Extensión recomendada

- 20-35 palabras por área.
- Total: 80-140 palabras.

### Ejemplos de áreas genéricas

```txt
Applied AI
Software Engineering
Data Systems
Product Building
Research
Design Engineering
Automation
Analytics
```

### Reglas

- Cada área debe tener una función clara.
- No convertir la sección en una lista enorme de tecnologías.
- No repetir exactamente los proyectos.

---

## 7. Proyectos destacados

### Objetivo

Demostrar capacidad real mediante evidencia concreta.

Es la sección más importante de un portfolio técnico o profesional.

### Número recomendado

- Mínimo: 3 proyectos.
- Ideal: 4 proyectos.
- Máximo en home: 5 proyectos.
- El resto puede ir en una página secundaria o GitHub.

### Estructura de cada proyecto

Cada proyecto debería tener:

```txt
Nombre del proyecto
Descripción breve
Problema
Qué se construyó
Tecnologías / herramientas
Resultado o estado
Links
```

### Formato recomendado de tarjeta

```txt
## Project Name

One-liner:
Descripción de una frase sobre qué es el proyecto.

Problem:
Qué problema resuelve o qué necesidad aborda.

Built:
Qué se diseñó, desarrolló o implementó.

Stack:
Tecnologías principales utilizadas.

Status / Result:
Estado actual, resultado medible o aprendizaje relevante.

Links:
GitHub · Demo · Case Study
```

### Extensión recomendada por proyecto

Versión corta:

- 70-120 palabras por proyecto.

Versión media:

- 120-180 palabras por proyecto.

Versión extensa, si tiene página propia:

- 400-800 palabras por proyecto.

### Orden interno recomendado

Ordenar proyectos por relevancia, no necesariamente por fecha.

Criterios de orden:

```txt
1. Más alineado con el perfil deseado
2. Más completo técnicamente
3. Más demostrable
4. Más reciente
5. Más diferencial
```

### Tipos de links

Cada proyecto puede incluir:

```txt
GitHub
Live demo
Video demo
Case study
Technical report
Presentation
Blog post
```

### Reglas

- No incluir demasiados proyectos mediocres.
- Mejor 3 proyectos bien explicados que 10 proyectos superficiales.
- Cada proyecto debe demostrar una competencia distinta o una evolución clara.
- Evitar descripciones genéricas sin evidencia.
- Si el proyecto está incompleto, indicarlo con claridad.

---

## 8. Página individual de proyecto

### Objetivo

Profundizar en proyectos especialmente importantes.

No todos los proyectos necesitan página propia. Solo los más fuertes.

### Estructura recomendada

```txt
1. Título del proyecto
2. Resumen ejecutivo
3. Contexto / problema
4. Objetivos
5. Solución construida
6. Arquitectura o metodología
7. Stack tecnológico
8. Retos técnicos
9. Resultado
10. Aprendizajes
11. Links y recursos
```

### Extensión recomendada

- Proyecto simple: 400-700 palabras.
- Proyecto técnico fuerte: 800-1.500 palabras.
- Proyecto con imágenes, diagramas o resultados: puede ser más largo.

### Formato sugerido

```txt
# Project Name

## Summary
Resumen de 3-5 líneas.

## Problem
Descripción del problema.

## Approach
Cómo se abordó.

## Architecture / Method
Componentes principales, flujo o metodología.

## Stack
Tecnologías usadas.

## Challenges
Problemas encontrados y decisiones tomadas.

## Result
Resultado, demo, métricas o estado actual.

## Learnings
Qué se aprendió.

## Links
Repositorio, demo, documentación o presentación.
```

### Reglas

- Incluir imágenes si ayudan: capturas, diagramas, arquitectura, outputs.
- Explicar decisiones, no solo funcionalidades.
- Mostrar trade-offs.
- No inflar resultados.
- Diferenciar claramente entre prototipo, MVP, producto terminado o trabajo académico.

---

## 9. Experiencia / trayectoria

### Objetivo

Mostrar recorrido académico, profesional o extracurricular.

### Formato recomendado

Timeline o lista vertical.

```txt
[Rol / etapa]
[Organización / institución]
[Fecha]
[Descripción breve]
[2-3 bullets de impacto o responsabilidades]
```

### Extensión recomendada

- 3-5 entradas.
- 40-80 palabras por entrada.
- Total: 200-350 palabras.

### Ejemplo de estructura genérica

```txt
### Role / Program / Degree
Organization · Date

Brief description of the role, context or program.

- Responsibility or achievement.
- Tool, method or area used.
- Result, output or learning.
```

### Reglas

- Priorizar experiencias relevantes.
- No duplicar todo LinkedIn.
- No incluir experiencias irrelevantes si no aportan al posicionamiento.
- Usar fechas de forma consistente.

---

## 10. Educación

### Objetivo

Dar contexto académico cuando sea relevante.

### Ubicación

Puede ser:

```txt
Opción A: dentro de Experience
Opción B: sección independiente
Opción C: breve bloque dentro de About
```

### Formato recomendado

```txt
Degree / Program
Institution
Dates
Relevant coursework / thesis / academic focus
```

### Extensión recomendada

- 1-3 entradas.
- 30-70 palabras por entrada.

### Reglas

- Incluir asignaturas solo si refuerzan el perfil.
- No convertirlo en expediente completo.
- Añadir tesis, proyectos o líneas académicas si son relevantes.

---

## 11. Habilidades / Skills

### Objetivo

Permitir escaneo rápido de competencias.

### Formato recomendado

Agrupar por categorías, no hacer una lista plana enorme.

```txt
Programming
- Tool 1
- Tool 2
- Tool 3

Data / AI
- Tool 1
- Tool 2
- Tool 3

Web / Product
- Tool 1
- Tool 2
- Tool 3

Other
- Tool 1
- Tool 2
- Tool 3
```

### Número recomendado

- 3-5 categorías.
- 3-7 elementos por categoría.
- 15-25 habilidades en total.

### Extensión recomendada

- Muy breve.
- 100-180 palabras como máximo.

### Reglas

- Separar habilidades reales de herramientas tocadas superficialmente.
- Evitar poner demasiadas tecnologías.
- Las skills deben estar respaldadas por proyectos o experiencia.
- Ordenar por relevancia.

---

## 12. CV / Resume

### Objetivo

Permitir descargar o consultar el CV completo.

### Formato recomendado

Botón visible en Hero o Contact:

```txt
Download CV
View Resume
```

### Ubicación recomendada

- Hero: si el CV es importante para oportunidades profesionales.
- Contact: como recurso final.
- Footer: como enlace secundario.

### Reglas

- Usar PDF.
- Mantener el archivo actualizado.
- Nombrar el archivo de forma profesional:

```txt
resume.pdf
cv.pdf
first-last-resume.pdf
```

---

## 13. Contacto

### Objetivo

Facilitar una acción concreta.

### Contenido recomendado

```txt
Email
LinkedIn
GitHub
Calendly, si aplica
Ubicación general, si aplica
Disponibilidad, si aplica
```

### Formato sugerido

```txt
## Contact

For opportunities, collaborations or questions, you can reach me here:

Email: [email]
LinkedIn: [link]
GitHub: [link]
```

### Extensión recomendada

- 40-80 palabras.
- 2-4 enlaces.
- Directo y claro.

### Reglas

- El email debe ser fácil de copiar.
- No esconder el contacto.
- No incluir demasiados canales.
- El formulario de contacto no es necesario para una primera versión.

---

## 14. Footer

### Objetivo

Cerrar la página con información secundaria.

### Contenido recomendado

```txt
Nombre
Año
Links secundarios
Repositorio de la web, si se quiere mostrar
```

### Formato sugerido

```txt
© [Year] [Name]. Built with [technology].
GitHub · LinkedIn · Email
```

### Extensión recomendada

- 1-2 líneas.
- No más de 30 palabras.

---

## 15. Extensión total recomendada

### Portfolio de una sola página

```txt
Hero: 60-90 palabras
About: 80-150 palabras
Focus Areas: 80-140 palabras
Projects: 300-700 palabras
Experience: 200-350 palabras
Skills: 100-180 palabras
Contact: 40-80 palabras
Footer: 10-30 palabras
```

Total aproximado:

```txt
870-1.720 palabras
```

Para una primera versión:

```txt
600-1.000 palabras
```

Para una versión más madura:

```txt
1.000-1.700 palabras
```

## 16. Estructura mínima viable

Esta es la estructura mínima para publicar rápido:

```txt
1. Hero
2. About
3. Projects
4. Skills
5. Contact
```

### Distribución

```txt
Hero: 1 bloque visual
About: 1 párrafo
Projects: 3 tarjetas
Skills: 3 grupos
Contact: email + 2 enlaces
```

### Archivos mínimos

```txt
index.html
styles.css
```

## 17. Estructura recomendada de archivos para HTML/CSS simple

```txt
portfolio/
├── index.html
├── styles.css
├── assets/
│   ├── images/
│   ├── icons/
│   └── documents/
│       └── cv.pdf
└── README.md
```

### Uso

```txt
index.html
Contenido y estructura de la web.

styles.css
Diseño visual.

assets/images
Fotos, capturas de proyectos, logos o diagramas.

assets/icons
Iconos si se usan.

assets/documents
CV u otros documentos descargables.
```

```txt

styles/
Define el sistema visual.
```

## 19. Jerarquía visual recomendada

### Orden de importancia visual

```txt
1. Nombre y posicionamiento
2. Proyectos destacados
3. Contacto
4. Experiencia
5. Skills
6. Footer
```

### Reglas

- El Hero debe ser claro, no decorativo.
- Los proyectos deben ocupar más espacio que las skills.
- Las skills deben ser escaneables.
- El contacto debe estar visible al final y, opcionalmente, también al inicio.
- Los títulos deben ser informativos.

---

## 20. Criterios de calidad

Antes de publicar, comprobar:

```txt
La web carga correctamente.
El nombre y rol se entienden en menos de 10 segundos.
Hay al menos 3 proyectos bien explicados.
Cada proyecto tiene problema, solución y tecnología.
Los enlaces funcionan.
El email es visible.
El diseño es legible en móvil.
No hay texto genérico innecesario.
No hay claims exagerados.
El CV está actualizado, si se incluye.
```

---

## 21. Errores frecuentes

Evitar:

```txt
Convertir la web en un CV largo.
Incluir demasiados proyectos sin explicación.
Usar frases genéricas sin evidencia.
Poner tecnologías que no aparecen en ningún proyecto.
Ocultar el contacto.
Diseño muy llamativo pero poco legible.
No adaptar a móvil.
No revisar enlaces.
No indicar el estado real de los proyectos.
Sobrecargar la home con texto.
```

## 23. Versión recomendada para primera publicación

```txt
Hero
About
Projects
Skills
Contact
Footer
```

Con esta distribución:

```txt
Hero: 20% del impacto visual
About: 15%
Projects: 40%
Skills: 15%
Contact/Footer: 10%
```

La prioridad debe ser publicar una versión simple, clara y verificable. El diseño y la complejidad técnica pueden mejorar después.
